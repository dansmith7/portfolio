# Ускорение загрузки на хостинге

## Что уже сделано в проекте

✅ **Preconnect к Supabase** — в `index.html` при сборке подставляется `<link rel="preconnect" href="...supabase.co">`, чтобы браузер заранее установил соединение с API.  
✅ **Разделение чанков** — React, Lenis, Supabase вынесены в отдельные бандлы; страницы (Work, About, ProjectDetail, админка) загружаются по мере перехода (lazy).  
✅ **Меньший первый экран** — главная (Home) грузится сразу; остальные страницы — по запросу.

---

## Что даёт хостинг

| Возможность | Эффект |
|-------------|--------|
| **CDN** | Статика (JS, CSS, шрифты, картинки) раздаётся с серверов рядом с пользователем → быстрее загрузка. |
| **Сжатие (gzip/brotli)** | Меньший объём файлов → быстрее передача. |
| **HTTP/2** | Параллельная загрузка ресурсов по одному соединению. |
| **Регион** | Хост ближе к пользователям и к региону Supabase → меньше задержка до API. |

---

## Рекомендации при выборе хостинга

1. **Близость к Supabase**  
   Если Supabase в `eu-central-1` — выбирайте хостинг в той же или соседней зоне (Европа). Так запросы к БД будут быстрее.

2. **CDN по умолчанию**  
   У Vercel, Netlify, Cloudflare Pages, GitHub Pages CDN включён. Статика раздаётся с края сети.

3. **Кэширование статики**  
   Хостинг должен отдавать заголовки вроде `Cache-Control: public, max-age=31536000` для JS/CSS (Vite при сборке добавляет хеши в имена файлов, поэтому длинный кэш безопасен).

---

## Быстрая проверка после деплоя

1. **Network (F12)**  
   Проверьте, что запросы к `*.supabase.co` идут сразу (preconnect помогает).  
2. **Lighthouse**  
   Запустите аудит производительности; смотрите FCP, LCP, TBT.  
3. **Регион**  
   Убедитесь, что хостинг и Supabase в одной/близкой зоне.

## Итог

Да, размещение на нормальном хостинге с CDN и сжатием ускорит загрузку. В проекте уже есть preconnect, разбиение бандлов и ленивая загрузка страниц — этого достаточно для хорошей скорости при деплое.
